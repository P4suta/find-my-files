Why it's fast

Not a watcher, not a crawler. It uses the structure of the filesystem itself.

⚡

Initial index from the $MFT

Reads the NTFS Master File Table directly — every file name on a volume in seconds.

🔄

Live via the USN journal

Change notifications come from the USN change journal. No file watchers, no rescans — updates reach the UI in about a second.

🧵

SIMD multithreaded scan

Multithreaded, SIMD substring scan over an in-memory index. p99 ≤ 50 ms at a million files, 3+ characters.

🪟

Native WinUI 3

Mica, a consistent dark theme, Per-Monitor V2 DPI — no blur on mixed-DPI setups.

🗜

Memory-lean by design

Indexing only file names means ~100 MB per million files — none of the multi-GB complaints content indexing brings.

🔐

Least privilege

MFT/USN access is split into a small Windows service. The UI runs unprivileged and talks over a locked-down, same-user-only named pipe.

What it deliberately does NOT do

Indexing file names only is the reason Everything is fast — and it's the core of find-my-files. Feature creep is a non-goal.

Out of scope (on purpose)

✕ Content search ✕ Property / tag indexing ✕ Previews ✕ FTP / HTTP servers ✕ FAT / exFAT / network drives ✕ ReFS ✕ Cross-platform

Built on "never crash, never hang, never go silent." The numbers the final milestone must hit.

Architecture

Default is an unprivileged UI plus the service. Without the service, it falls back to an elevated in-process engine.

WinUI 3 app (C#, unprivileged) ──named pipe──▶  fmf-service (Rust, LocalSystem)
   └─ IEngineClient boundary                       └─ fmf-core: $MFT scan, USN tailing,
       ├─ PipeEngineClient (default)                  in-memory index, query engine
       └─ FfiEngineClient ──P/Invoke──▶  fmf_engine.dll (elevated in-proc fallback)

Get started

Toolchain pinned via mise, tasks run with just.

mise install        # rust + dotnet
just setup          # toolchain + git hooks
just build          # engine (cargo, release)
just service-install  # register the service (elevated, once)

Prebuilt binaries are on the Releases page.
