なぜ速いのか

監視ツールでもクロールでもありません。ファイルシステムそのものの構造を直接使います。

⚡

$MFT 直読みで初回索引

NTFS の Master File Table を直接読み、1ボリュームあたり数秒で全ファイル名を取り込みます。

🔄

USN ジャーナルで即時反映

変更通知は USN チェンジジャーナルから。ファイル監視も再スキャンも不要で、変更が UI に約1秒で届きます。

🧵

SIMD マルチスレッド検索

インメモリ索引上をマルチスレッド + SIMD で部分一致走査。100万件・3文字以上で p99 ≤ 50ms。

🪟

ネイティブ WinUI 3

Mica、一貫したダークテーマ、Per-Monitor V2 DPI。混在DPI環境でもにじみません。

🗜

省メモリ設計

ファイル名だけを索引するから、100万件で約100MB。content索引が生んだ「メモリ8GB級」の苦情とは無縁です。

🔐

権限を最小化

MFT/USN 読み取りの特権は小さな Windows サービスに分離。UI は非特権で、施錠された名前付きパイプ(同一ユーザー限定)で通信します。

あえて、やらないこと

ファイル名だけを索引する——これが Everything が速い理由であり、find-my-files の核です。機能の盛り込みは非目標です。

スコープ外（意図的に）

✕ content検索 ✕ プロパティ・タグ索引 ✕ プレビュー ✕ FTP / HTTP サーバ ✕ FAT / exFAT / ネットワークドライブ ✕ ReFS ✕ クロスプラットフォーム

性能の合格ライン

「落ちない・固まらない・黙らない」を前提に、最終マイルストーンで満たす数値目標。

指標目標
初回インデックス（実C:）25万 ≈ 5s / 100万 ≈ 60s
検索 p99（100万件・3文字以上）≤ 50 ms
変更反映（USN → UI）≤ 1 s
スナップショット復元 → ready≤ 2 s
RAM（エンジン単体・バイト/ファイル）≤ 110 B

アーキテクチャ

既定は非特権UI + サービス。サービス未導入時は昇格in-procフォールバック。

WinUI 3 app（C#・非特権） ──named pipe──▶  fmf-service（Rust・LocalSystem）
   └─ IEngineClient 境界                       └─ fmf-core: $MFT走査・USN tailing・
       ├─ PipeEngineClient（既定）                  インメモリ索引・クエリエンジン
       └─ FfiEngineClient ──P/Invoke──▶  fmf_engine.dll（昇格in-procフォールバック）

はじめる

ツールチェーンは mise で固定、タスクは just。

mise install        # rust + dotnet
just setup          # toolchain + git hooks
just build          # エンジン（cargo, release）
just service-install  # サービス登録（昇格・一度だけ）

ビルド済みバイナリは Releases から。
